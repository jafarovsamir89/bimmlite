## 1. Видение и цели

**Что делаем:** серверная система диагностики/кодирования BMW + тонкий клиент-мост.

**Ключевые принципы (чего НЕ повторяем из bimmerapp):**

- Никаких дублирующихся API, сервисов и баз данных — один источник правды на каждую сущность.
- Вся «моторика» (декод, ключи, БД, AI, кодирование) — на сервере. Клиент максимально тонкий.
- Всё логируется — от старта до продакшена. Любой шаг виден в логе.
- Модульность и чистые границы слоёв с самого начала.

**Функционал (MVP → далее):**

- [ ]  Определение авто по ENET-кабелю (позже: Android Wi-Fi / Bluetooth)
- [ ]  Чтение ошибок (DTC) по всем блокам
- [ ]  AI-анализ ошибок
- [ ]  Чтение стандартных параметров ЭБУ
- [ ]  Кодирование (FDL)
- [ ]  Аккаунты + лицензирование (подписка)
- [ ]  Сильное сквозное логирование
- [ ]  Красивый UI и удобный UX

## 2. Архитектура: 3 слоя

<aside>
🧩

Главная идея: **«тонкий мост + умный сервер»**. Мост НЕ содержит бизнес-логики, ключей и баз — только исполняет транспорт. Это защищает IP (RSA-ключи и логика кодирования никогда не покидают сервер), включает контроль лицензий и делает клиент легко переносимым на Android.

</aside>

### Слой 1 — Bridge (клиент-мост)

- Обнаруживает машину и держит физический канал (ENET сейчас; Wi-Fi/BT позже).
- Устанавливает защищённый туннель к серверу (WebSocket/gRPC + TLS + токен сессии).
- Исполняет **пакеты команд** от сервера с локальными тайминг-гарантиями (см. §5).
- Локальный буфер логов + отправка на сервер.
- Платформы: Desktop (Windows/mac/Linux) сначала, потом Android.

### Слой 2 — Backend (ядро)

- Оркестрация всех операций: connect, scan ECU, read DTC, read params, coding.
- Декодирование CAFD/FDL, пересчёт NCD-подписи, SecurityAccess (RSA-ключи).
- Аккаунты, лицензии, подписки, права доступа.
- AI-сервис по ошибкам.
- Централизованное логирование и трейсинг.

### Слой 3 — Данные и AI

- БД расшифрованных кодировок (`ecu_logic`), БД ошибок (DTC), RSA-ключи, SALAPA-маппинг — мигрируем из bimmerapp, но чисто.
- Пользовательская БД (аккаунты, подписки, сессии, аудит-логи).
- AI-провайдер (OpenAI/OpenRouter).

```
[BMW ECU] ⇄ ENET/WiFi/BT ⇄ [Bridge client]  ⇄ TLS-туннель ⇄  [Backend]
                                 (только транспорт)        (вся логика, ключи, БД, AI)
                                                                 │
                                                     [PostgreSQL] [Data blobs] [AI] [Logs]
```

## 4. Уроки bimmerapp (что чистим)

<aside>
⚠️

В bimmerapp было: 3 провайдера SecurityAccess, несколько путей чтения VO, дублирующиеся форматы DID, разрозненные бэкапы, смешение слоёв. В BimmLite — по одному каноничному пути на операцию.

</aside>

- **Один** SecurityAccess-модуль с провайдерами через единый интерфейс (не 3 файла).
- **Один** транспортный клиент UDS (bimmerapp имел логику в uds_service + fdl_reader + dtc_reader + ecu_scanner — консолидируем).
- **Одна** схема данных ЭБУ (никаких параллельных `sa_keys_l3` / `sa_keys_v2` без причины — мигрируем в единый store).
- **Один** формат бэкапа/снимка.
- Чистое разделение: `api` → `services` → `data`, без бизнес-логики в роутерах.

## 5. Транспорт и тайминг (критично для кодирования)

<aside>
⏱️

UDS чувствителен к таймингам (S3-таймер, NRC 0x78 ResponsePending, задержки SecurityAccess). Гонять каждый байт через интернет опасно.

</aside>

**Решение — «командные пакеты»:** сервер отправляет мосту не отдельные байты, а декларативный сценарий шага (например: «войти в сессию 0x03 → TesterPresent → запросить seed»). Мост исполняет локально с жёсткими таймингами и возвращает результаты (в т.ч. seed). Сервер вычисляет key (ключи остаются на сервере) и шлёт следующий пакет.

- Один сетевой round-trip на логический шаг, а не на кадр.
- Ключи и логика подписи — только на сервере.
- Мост держит TesterPresent keepalive локально по инструкции сервера.
- Все кадры (hex) логируются на обеих сторонах с общим trace_id.

## 6. Логирование и наблюдаемость (приоритет №1)

<aside>
🔍

Требование: любой шаг виден в логе. Не подключилось — видно почему. Не кодируется — видно почему. Это закладывается в фундамент с первого дня, а не прикручивается потом.

</aside>

**Принципы:**

- **Структурные логи** (JSON): timestamp, level, module, event, trace_id, session_id, user_id, vin, ecu, duration_ms, payload_hex, result, error.
- **Correlation ID (trace_id)** сквозной: клиент → мост → сервер → машина → и обратно. Одна операция = один trace.
- **Уровни:** TRACE (каждый UDS-кадр) / DEBUG / INFO (шаги операции) / WARN / ERROR / CRITICAL.
- **Каждый шаг операции — событие:** `connect.discover.start`, `connect.vin.ok`, `security.seed.received`, `security.key.rejected(NRC 0x35)`, `coding.block.write.fail(NRC 0x31)` и т.д.
- **Двусторонний сбор:** мост шлёт свои логи на сервер (буфер + дозапись), чтобы видеть проблемы на стороне клиента/машины.
- **Аудит-лог** отдельно: кто/когда/какое кодирование применил (для лицензий и отката).
- **Хранение:** БД + файлы; готовность к внешним системам (OpenTelemetry/Grafana/Sentry) без переписывания кода.
- **UI логов:** просмотр логов сессии в приложении (live-стрим шагов операции).

**Пример строки лога:**

```json
{"ts":"2026-07-04T00:00:00Z","level":"ERROR","module":"security","event":"key.rejected","trace_id":"a1b2","session_id":"s-77","user_id":"u-3","vin":"WBA...","ecu":"KOMBI","nrc":"0x35","msg":"invalidKey — check magic/RSA for BL_ID"}
```

## 7. Аккаунты и лицензирование

- Регистрация/логин, роли (user/admin), устройства.
- Подписки/тарифы: лимиты по операциям (например, кодирование только на платном тарифе).
- Проверка лицензии **на сервере** перед каждой чувствительной операцией (кодирование, SecurityAccess).
- Все проверки логируются (почему отказано в операции — видно в логе).
- Клиент без валидной подписки = только просмотр/диагностика (настраиваемо).

## 11. Открытые решения (нужно подтвердить)

**Подтверждено (04.07.2026):**

- Репозиторий: **монорепо с раздельными пакетами** (`backend/`, `bridge/`, `frontend/`, `shared/`, `data/`) — один репо, чистые границы.
- Desktop-мост: **Go** (лёгкий бинарник, простой кросс-компайл под Windows/mac/Linux).
- База данных: **PostgreSQL** (аккаунты, логи, подписки, аудит).
- Backend: **Python + FastAPI** (переиспользуем логику/ключи/БД из bimmerapp).

**Финальные технические решения:**

- Протокол: **сервер собирает точные UDS-кадры**, мост — «тупая труба» с тайминг-правилами (отправь байты → жди ответ → повтор при NRC 0x78 → TesterPresent keepalive). Вся логика UDS/DoIP/HSFZ — только на сервере.
- Туннель мост↔сервер: **WebSocket + TLS** (дуплекс, стриминг логов).
- Хостинг: **Google Cloud**.
- Логи: **встроенный просмотрщик логов в приложении** (структурные JSON-логи в БД + live-стрим шагов операции; архитектура готова к внешним системам позже).

## 13. Технологический ландшафт (изучено в исследовании)

<aside>
🔬

Глубокий анализ всех похожих проектов (04.07.2026). Главный вывод: **никто из открытых/массовых продуктов не делает server-native архитектуру** (логика на сервере + тонкий мост). Существующий «remote coding» — это TeamViewer-стиль (человек удалённо управляет E-Sys), а не наша модель. Наш подход — реальный дифференциатор.

</aside>

**Профессиональные (BMW OEM-уровень):**

- **ISTA / ISTA-P** — дилерская диагностика+программирование (E/F/G). Эталон по покрытию.
- **E-Sys + PSdZData** — инженерное кодирование (FDL/VO) + флеш. Требует CAFD-mapping (TokenMaster/E-Sys Launcher).
- **NCS Expert / WinKFP / INPA** — legacy для E-серии.

**Массовые приложения (наши прямые конкуренты):**

- **BimmerCode** — кодирование, чистый UI, разовая оплата, WiFi/BT-адаптеры (не Ethernet на телефоне).
- **BimmerLink** — диагностика + live-данные + сервисные функции (регистрация АКБ и т.д.).
- **Carly** — подписка, широкий охват, свой адаптер (модель монетизации — ориентир для нас).
- **xHP / xAutomotive** — тюнинг АКПП (TCU) через телефон+WiFi-ENET; модель лицензий по VIN и «stage» — хороший пример.

**Открытые проекты (референсы):**

- **EdiabasLib / Deep OBD** (GPL) — мультиадаптерность (ENET/ELM327/D-CAN/BT/WiFi), Android. Эталон для нашего adapter-слоя.
- **Caring Caribou** (GPL) — DoIP/UDS discovery, dump DIDs, security_seed, tester_present. Референс для модуля обнаружения.
- **scapy** `contrib/automotive/bmw/hsfz.py` — готовая справка по HSFZ (control words 0x01/0x02/0x10–0x13…).
- **TokenMaster** — CAFD/NCD mapping и токены (понимание того, как устроено именование параметров).

## 14. Открытые библиотеки и стандарты (переиспользуем вместо велосипеда)

<aside>
🛠️

Важная находка: в bimmerapp UDS/DoIP писались с нуля (много багов). Есть зрелые библиотеки под MIT — можно опереться на них на сервере для сборки/разбора кадров.

</aside>

- **UDS:** `python-udsoncan` (MIT, ISO 14229) — сборка/разбор UDS на сервере.
- **DoIP:** `python-doipclient` (MIT, ISO 13400) + `udsonip` (связка для multi-ECU).
- **Go:** `go-uds` (порт udsoncan) — если понадобится логика на стороне моста (но у нас мост — тупая труба, так что не критично).
- **HSFZ (BMW):** справка из scapy; это не покрыто udsoncan — наш собственный слой (знание из bimmerapp — чисто).
- **Лицензии:** udsoncan/doipclient = **MIT** (можно в проприетарный продукт). Caring Caribou / EdiabasLib = **GPL** — только как референс, код не копируем.

**Профессиональные стандарты (уровень «sупер-инженера»):**

- **ISO 14229** (UDS), **ISO 13400** (DoIP; TLS на порту 3496 в 2019/2025), **HSFZ** (BMW, порт 6801, OBD pin 8 = EthernetActivationLine).
- **ODX (ISO 22901) / OTX (ISO 13209) / PDX** — профессиональный XML-формат описания диагностики/DTC. Каталог можно спроектировать ODX-совместимо.
- **MVCI / D-Server (ISO 22900)** — наш backend по сути есть кастомный D-Server. **J2534** — pass-through стандарт (на будущее).

## 15. ПРОБЕЛЫ: что добавляем в архитектуру (по итогам research)

<aside>
✅

Сравнение нашего документа с лучшими практиками выявило 10 дополнений. Они включаются в план.

</aside>

1. **VO/FA-кодирование как отдельная операция** (не только FDL). VO = конфигурация авто (SALAPA/FA) → сброс ЭБУ к дефолтам; FDL = точечные параметры. Нужны обе.
2. **I-Step awareness:** валидность кодировок зависит от Integration Step авто; версия данных ≥ I-Step машины.
3. **Граница scope:** MVP = диагностика + кодирование. **Программирование/флеш ECU (ISTA-P/WinKFP) — ВНЕ MVP** (высокий риск кирпича).
4. **Протокольная вариативность:** F-серия = HSFZ/ENET (6801); G-серия/новее = стандартный DoIP (13400, TLS 3496). Явно: discovery → routing activation → diagnostic messages.
5. **Adapter abstraction в мосте с 1 дня:** единый интерфейс под ENET / ENET-over-WiFi / BT / ELM327 (с учётом, что ELM327 не тянет полное кодирование).
6. **Новые категории функций:** **live-данные** (PID-стриминг сенсоров, BimmerLink-стиль) и **сервисные функции** (регистрация АКБ, сбросы CBS, адаптации, DPF-регенерация).
7. **Лицензирование глубже:** device fingerprint / node-lock моста + entitlements по VIN/операции + offline-grace токены. Server-native = внутренне антипиратская модель.
8. **Наблюдаемость — 3 столпа:** к логам и трейсам добавить **metrics** (процент успешных подключений/кодирований, частота NRC-ошибок) — для роли «мозга-надзора».
9. **Каталог — ODX-совместимая модель:** рассмотреть ODX-подобную схему для DTC/параметров (профессионально и расширяемо).
10. **Правовой/этический нота:** SecurityAccess/токены — серая зона; server-native держит ключи на сервере (безопаснее юридически и технически).

## 16. Подтверждённые сильные стороны (валидировано рынком)

- **Server-native = уникальный дифференциатор:** конкуренты держат логику/данные на клиенте. Наш подход защищает IP, упрощает лицензирование и обновления.
- **Сильное логирование с первого дня** — совпадает с best practice «observability from day one».
- **Тонкий мультиплатформенный мост** — лёгкий переход на Android Wi-Fi/BT (как xHP/BimmerCode, но без логики на клиенте).

### Ключевые источники

- python-udsoncan (ISO 14229, MIT) · python-doipclient (ISO 13400, MIT)
- scapy HSFZ (BMW) · HSFZ/DoIP разбор (dissecto)
- Caring Caribou (DoIP-модуль) · EdiabasLib / Deep OBD
- ODX ISO 22901 (Vector) · ODX/OTX/PDX для диагностики
- VO vs FDL coding · DoIP remote diagnostics в облаке
